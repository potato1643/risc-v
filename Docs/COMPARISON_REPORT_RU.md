# Сравнительный анализ тестовых наборов RISC-V

## 1. Архитектура тестового стенда

Тестовый стенд развёрнут в двух контейнерах Docker.

| Контейнер | Архитектура | Назначение |
|-----------|-------------|------------|
| `riscv-env` | ARM64 | RISCOF + MicroTESK |
| `compiler-lab` | x86-64 | Imperas + сборка Sail |

**Ключевой принцип:** все три тестовых набора запускаются на **одном и том же** симуляторе Spike, скомпилированном с флагами `--coverage` (GCC). Это позволяет проводить прямое сравнение покрытия кода симулятора.

Конвейер: `① Запуск тестов → ② gcov/lcov → ③ Сравнительный анализ`

---

## 2. Единая методика сбора покрытия кода (gcov/lcov)

1. Очистка счётчиков gcov: `find $BUILD_DIR -name "*.gcda" -delete`
2. Последовательный запуск всех ELF-файлов на инструментированном Spike
3. Сбор данных: `lcov --capture --directory $BUILD_DIR --output-file <suite>_coverage.info`
4. Генерация HTML-отчёта: `genhtml`

| Расширение | Описание | Статус |

## 7. Выводы

1. **Единая методика работает.** Все три набора запущены на одном gcov-инструментированном Spike, получены сравнимые метрики. MicroTESK — наивысший Line Coverage (16.2%).

2. **Объединённое покрытие — 16.9%** (+0.7 п.п. к лучшему отдельному). Высокое перекрытие (~92%) — наборы тестируют во многом одни и те же участки Spike. Но каждый набор даёт 78–244 строки уникального покрытия.

3. **Покрытие — только расширение I.** 15–17% Line Coverage отражают тестирование лишь базовых целочисленных инструкций. Расширения M, A, C, F, D и др. (28+) не протестированы на gcov-Spike. Каждое активирует отдельные модули Spike — их запуск является прямым путём к повышению покрытия.

4. **RISCOF — наиболее полный по ISA-покрытию.** 100% 21 052 точек RV64I+M (CGF). Рекомендуется как основной инструмент для расширения gcov-тестирования на другие расширения.

5. **Imperas полезен для кросс-валидации и 32-битных путей.** Выявлено одно расхождение с riscvOVPsim (допустимое спецификацией). RV32I-тесты покрывают 32-битные пути Spike (+242 уникальные строки).

6. **MicroTESK дополняет поинструкционные тесты.** Алгоритмические тесты задействуют больше ветвей кода и дают наивысший Line Coverage.
