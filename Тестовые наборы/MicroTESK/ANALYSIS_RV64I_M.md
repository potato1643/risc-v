## 2. Правило: пользовательский vs системный режим по расширениям

При анализе инструкционального охвата разделение на пользовательский/системный
режим **актуально только для базового расширения I (RV32I/RV64I)**, поскольку
оно охватывает оба уровня привилегий. Остальные расширения являются либо
чисто пользовательскими, либо чисто системными.

### Классификация расширений

| Расширение | Режим | Обоснование |
|-----------|:---:|------|
| **I** (RV32I/RV64I) | Оба | Базовый ISA: ADD/SUB/LW/SW/... (польз.) + ECALL/EBREAK/CSR (систем.) |
| **M** (умножение/деление) | Только польз. | Все инструкции работают с x0-x31. ECALL/MRET в данных тестирования — это инфраструктура Spike (выход из программы), не часть M-расширения. |
| **A** (атомарные) | Только польз. | LR/SC/AMO оперируют памятью через x-регистры |
| **C** (сжатые) | Только польз. | 16-битные кодировки обычных пользовательских инструкций |
| **F** (FP32) / **D** (FP64) | Только польз. | Работают с f0-f31, без CSR |
| **Zicsr** | Только систем. | CSR read/write — привилегированная архитектура |
| **Zifencei** | Только систем. | FENCE.I — сброс кэша инструкций |
| **MRET/SRET/WFI и др.** | Только систем. | Привилегированная спецификация |

### Простое правило

```
Инструкция оперирует x0-x31 или f0-f31  →  ПОЛЬЗОВАТЕЛЬСКИЙ РЕЖИМ
Инструкция оперирует CSR или меняет уровень привилегий  →  СИСТЕМНЫЙ РЕЖИМ
```

### Следствие для анализа MicroTESK

- **RV64I** → нужно разделять пользовательский и системный режим (72 инструкции: 62 польз. + 10 систем.)
- **RV64M** → анализируется как чисто пользовательский набор (13/13 = 100%)
  - ECALL (13 вызовов) и MRET (13 вызовов) в данных RV64M — это фиксированные затраты на запуск/выход каждой тестовой программы через прокси-ядро Spike, **не являются частью M-расширения**

## 4. RV64M: посегментный анализ (13 инструкций, только пользовательский режим)

> **Почему без разделения на пользовательский/системный?**
> Все 13 инструкций M-расширения (MUL, DIV, REM и их варианты) оперируют
> исключительно регистрами x0-x31 — это чисто пользовательский режим.
> Наблюдаемые в данных ECALL (×13) и MRET (×13) — это инфраструктурные
> вызовы прокси-ядра Spike при запуске и выходе каждой тестовой программы,
> они не относятся к M-расширению.

## 5. Ключевые находки и пробелы

### 5.1 Сдвиги — полное отсутствие (12 инструкций)

Ни одна из 12 инструкций сдвига не используется в тестах MicroTESK RV64I:
```
SLL, SRL, SRA          — регистровые сдвиги (RV32I)
SLLI, SRLI, SRAI       — непосредственные сдвиги (RV64I 6-битный вариант)
SLLW, SRLW, SRAW       — регистровые сдвиги слов (RV64I)
SLLIW, SRLIW, SRAIW    — непосредственные сдвиги слов (RV64I)
```

### 5.2 Беззнаковые операции — полное отсутствие (7 инструкций)

```
BLTU, BGEU     — беззнаковые условные переходы
LBU, LHU, LWU  — беззнаковая загрузка из памяти
SLTU, SLTIU    — беззнаковое сравнение
```

### 5.3 JAL — не используется

MicroTESK использует только JALR (30 выполнений) для возврата из функций.
Для сравнения: RISCOF — 61,447 вызовов JAL, Imperas — 66,452 вызовов JAL.
Это указывает на то, что MicroTESK генерирует тесты без использования
pc-относительного вызова подпрограмм.

### 5.4 M-расширение — идеальное покрытие

В отличие от RV64I, RV64M покрыт на 100%. MicroTESK включает беззнаковые
варианты умножения/деления/остатка (DIVU, REMU, MULHU, MULHSU), которые
отсутствуют в RV64I тестах.

## 6. Сравнение RV64I: пользовательский vs системный режим

| Аспект | Пользовательский | Системный |
|--------|:---:|:---:|
| Инструкций | 62 (#1–52 + #53) | 10 (#54–62 + #63–72) |
| Покрыто | 30 | 4 |
| Процент | 48.4% | 40.0% |

**Системные инструкции MicroTESK, получившие ненулевое покрытие:**
- CSRRW (420) — атомарное чтение-запись CSR
- CSRRS (242) — атомарное чтение-установка битов CSR
- CSRRWI (300) — немедленное атомарное чтение-запись CSR
- MRET (30) — возврат из машинного режима

Системные инструкции ECALL (29) и MRET (30) используются инфраструктурой
Spike (прокси-ядро), а не генерируются тестовыми шаблонами напрямую.
CSR-инструкции (CSRRW/CSRRS/CSRRWI) активно генерируются тестовыми шаблонами.

## 7. Сравнение RV64I vs RV64M: контраст в покрытии

| Характеристика | RV64I | RV64M |
|--------|:---:|:---:|
| Инструкций | 72 | 13 |
| Покрыто | 35 | 13 |
| Процент | 48.6% | 100.0% |
| Беззнаковые покрыты | 0/7 | 1/1* |
| Сдвиги покрыты | 0/12 | N/A |

> *M-расширение включает беззнаковые варианты (DIVU, REMU, MULHU, MULHSU),
> и все они покрыты.

Разница в качестве покрытия (48.6% vs 100%) указывает на то, что шаблоны
RV64M (compliance/rv64um) были разработаны более систематично, чем шаблоны
для RV64I (examples + algorithms + synthetics).

1. **Добавить шаблоны сдвигов** — создать тесты для SLL/SRL/SRA и их вариантов
2. **Добавить беззнаковые операции** — BLTU/BGEU/LBU/LHU/LWU/SLTU/SLTIU
3. **Добавить JAL** — тесты с pc-относительными вызовами подпрограмм
4. **Добавить RV64I W-операции** — ADDW/SUBW/SLLW/SRLW/SRAW
5. **Расширить CSR-тесты** — покрыть CSRRC/CSRRSI/CSRRCI
